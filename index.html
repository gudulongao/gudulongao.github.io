垃圾收集与内存分配策略

发表于 2019-10-28 | 分类于 jvm |

垃圾回收

垃圾收集GC(Garbage Collection)。JVM中的程序计数器，虚拟机栈，本地方法栈随着线程而生，线程而灭，栈中的栈帧随着方法的进入和退出有条不紊的执行着入栈和出栈的操作。每一个栈帧中分配的内存大小基本在类结构确定下来时已知。所以这几个区域内就不需要过多的考虑内存回收问题，因为方法结束或者线程结束时，内存自然就跟着回收了。而Java堆和方法区则不一样，一个接口中的多个市县累需要的内存可能不一样，一个方法中的多个分支需要的内存也可以不一样。我们只有在程序处于运行期间时才知道会创建哪些对象，这些内存的分配和回收都是动态的。垃圾收集器所关注的是这部分内存。

垃圾回收主要是针对堆和方法区

对象已死

引用计数算法

引用计数（Reference Counting）：给对象中添加一个引用计数器，每当有一个地方引用时，计数值加1；当引用失效时，计数器减1。任何计数器为0的对象是不可能被使用的。

主流的虚拟机未采用，因为无法解决循环引用的问题。

可达性分析算法

可行性分析（Reachability Analysis）：通过一系列成为“GC Roots”的对象作为起始点，从这些节点开始向下搜索，搜索所走过的路径成为引用链。当一个对象到GC Roots没有任何引用链相连接时，证明此对象是不可用的。

在JVM中 可以作为GC Roots的对象有：

  • 虚拟机栈中引用的对象（栈帧中的本地变量表）
  • 方法区中类静态属性引用的对象
  • 方法区中常量引用的对象
  • 本地方法栈中JNI(本地方法)引用的对象

引用

类型 说明 实现类
强引用 类似Object obj = new Object() 这类的引用，只要强引用还在，垃圾收集器永远不会回收被引用的对象
软引用 描述一些还有用但并非必须的对象，在系统将要发生内存溢出异常之前，将软引用的对象列进回收范围之中进行第二次回收。如果这册回收还没有足够的内存，才会抛出内存溢出异常。 SoftReference
弱引用 比软引用的强度更弱。弱引用关联的对象只能存活到下一次垃圾收集发生之前。当垃圾收集器工作时，无论当前内存是否足够，都会回收掉弱引用的对象 WeakReference
虚引用 最弱的引用关系，一个对象是否有需引用的存在，完全不会对其生存时间构成影响，也无法通过虚引用来取得一个对象实例。虚引用的唯一作用是在这个对象被垃圾收集器回收时收到一个系统通知 PhantomRefernece

回收

两次标记：

  • 可达性分析后没有与GC Roots相连的引用链，进行第一次标记并进行筛选。筛选的条件时判断是否有必要执行finalize()方法。当对象没有覆盖finalize方法，或者finalize方法已经被jvm执行过都被判断为没有必要执行。
  • 如果判断为有必要执行finalize。对象将会进入F-Queue队列中。由JVM自动建立的低优先级的Finalizer线程来消费队列中的对象，消费的动作就是执行对象的finalize方法。（并不会等待finalize执行完毕，避免有些对象的finalize的执行逻辑缓慢或者死循环导致整个F-Queue中的其他对象一直等待）。稍后GC会对F-Queue中的对象进行第二次标记。如果对象想要拯救自己，只能在finalize中将自己与引用链中的任何一个对象产生关系即可。

方法区回收

方法区的回收主要是针对：废弃的常量和无用的类。

可回收类的条件：

  • 该类所有的实例都已回收，即jvm堆中不存在该类的任何实例。
  • 加载该类的classloader已经被回收
  • 该类对应的java.lang.Class对象没有在任何地方被引用，无法在任何地方通过反射调用该类的方法。

可回收常量的条件：常量池中数据没有任何其他对象锁引用。

以字符串常量池中的字符串”abc”为例。当系统中不再有任何一个字符串内容是”abc”后，常量池中的字符串”abc”可以被回收。常量池中的其他类，接口，方法，字段符合引用都类似。

垃圾回收算法

标记-清除算法

标记-清除算法（Mark-Sweep）：

  • 标记：标记出所有要回收的对象。
  • 清除：统一回收标记的对象。

优点

后续垃圾收集算法的基础。简单易懂

缺点

  • 效率问题：标记和清除两个操作的效率都很低
  • 空间问题：清除后容易产生零碎的内存空间，空间碎片太多可能导致后续程序运行期间需要分配大对象时，无法找到足够的连续内存而不得不触发一次垃圾收集动作。

复制算法

复制算法（Copying）：将内存空间分割成大小相等的两块，每次只使用其中的一块，当一块用完了，就将还存活的对象复制到另一块，然后将使用过的空间整个清空。

优点

实现简单，不会产生内存碎片

缺点

  • 内存空间利用率只有原先的一半。
  • 对象存活率高的时候需要较多次的复制操作，效率低。

标记-整理算法

标记-整理算法（Mark-Compact）:标记过程与标记-清除算法一样，只是标记后的对象不是立即就清除回收，而是让所有存货的对象都向一段移动，然后直接清理掉端边界以外的内存。

优点

  • 适合老年代（对象存活率高）

    缺点

  • 存活对象的移动 实现复杂

停止线程

线程正常运行完成之后会自动结束。但Thread类提供了废弃的stop方法来在外部干预来停止线程。

stop方法的停止过于粗暴，会使线程立即结束，为了防止线程在对数据的变更操作未完成就遭到停止，导致数据只被修改了一半，影响数据完整性，不建议使用该方法停止线程。

线程中断

线程中断也是通过外部方式让线程停止，但不同于stop，线程中断并不会使线程立即结束，只是给线程发一个通知，告知线程需要退出，线程收到通知之后可以自行处理是否退出。

Thread类提供了三个涉及中断操作的API：

等待(wait)和通知(notify)

Object类提供了wait和notify方法，来完成多线程之间的协作。

  • 在synchronized同步块中，线程拿到对象锁后，可以通过wait方法，将当前线程存放到对象锁的等待队列中。当前线程就会停止执行，并释放对象锁。
  • 当有线程拿到对象锁之后，可以通过notify方法，从对象锁中的等待队列中随机释放一个线程，并唤醒。也可以通过notify方法将对象锁等待队列中的所有线程都释放并唤醒。

特点

  • wait/notify/notifyAll 需要事先获取对象锁，即需要在synchronized同步代码块中。
  • wait执行后线程会释放对象锁，并停止执行。当notify释放该线程后，线程还是需要再次获取到对象锁之后，才能从wait后面的位置继续执行。
  • notify/notifyAll之后 线程正常执行完毕之后释放对象锁。

volatile

volatile：修饰符，用volatile修饰的变量不会进行指令重排。

特性

volatile修饰的变量，在多线程并发场景下，具备一下特征：

  • 可见性：一个线程对变量做的修改，其他线程立即可见。
  • 有序性：禁止指令重排。
  • 弱原子性：对于单个操作具备原子性，但对复合操作如++ 不具备原子性。

votaile并不能保证线程安全，只能保证当一个线程修改了数据之后，其他线程能够看到这个改动，但当两个线程同时修改同一个数据时，就会产生冲突

可见性实现逻辑

Lock指令

奔腾（Pentium）处理器等时期，在处理Lock指令时，会产生一个Lock信号，会对总线进行加锁，这个期间其他处理器便无法对内存进行读写操作。后续的处理器都采用高速缓存锁代替总线锁来处理。

缓存一致性

嗅探协议（snooping）：

  • 缓存是分段的，一段对应一块存储空间。是CPU缓存中可分配的最小内存单元。
  • 所有对内存中的数据传输都在总线上，所有的处理器都可以看到这条总线。
  • 缓存本身是独立的，但内存是共享的，所有内存操作都需要仲裁（同一个指令周期内只有一个CPU可以操作内存）
  • CPU缓存不只是在数据传输时与总线打交道，还在不停的“嗅探“其他缓存的数据变化。
  • 当一个CPU缓存中的数据发生变化时，其他处理器都会收到通知，来使得自己的缓存数据进行同步更新。

总结

  • 当处理器在处理涉及到volatile修饰的变量时，JVM会自动给CPU添加一个Lock指令。
  • CPU在处理Lock指令时，会产生一个Lock信号。这样会阻塞其他处理器对内存的读写操作。
  • CPU完成volatile的运算后，将结果存储到处理器缓存(L1,L2等)中对应的行缓存中。并立即将对应行缓存通过总线将数据同步到内存中。同时这一操作会产生一个数据变化的通知。
  • 其他处理器缓存会在总线上嗅探到数据变化，其他处理器会收到数据变化的通知，会立即将变化的内存数据同步到处理器缓存中。

有序性实现逻辑

内存屏障

volatile的可见性是基于内存屏障（Memory Barrier）实现的。内存屏障是一个CPU指令，程序在运行期间，编译器为了提高运行效率会对指令进行重排，MM 为了保证在不同的编译器和 CPU 上有相同的结果，通过插入特定类型的内存屏障来禁止特定类型的编译器重排序和处理器重排序，插入一条内存屏障会告诉编译器和 CPU：不管什么指令都不能和这条指令重排序。

内存屏障 说明
StoreStore屏障 禁止上面的普通写和下面的 volatile 写重排序。
StoreLoad屏障 防止上面的 volatile 写与下面可能有的 volatile 读/写重排序。
LoadLoad屏障 禁止下面所有的普通读操作和上面的 volatile 读重排序。
LoadStore屏障 禁止下面所有的普通写操作和上面的 volatile 读重排序。
  • 在每个 volatile 写操作的前面插入一个 StoreStore 屏障。
  • 在每个 volatile 写操作的后面插入一个 StoreLoad 屏障。
  • 在每个 volatile 读操作的后面插入一个 LoadLoad 屏障。
  • 在每个 volatile 读操作的后面插入一个 LoadStore 屏障。

应用场景

适合采用volatile的场景必须满足以下两个条件：

  • 对变量的写操作不依赖于读。
  • 变量没有包含在其他变量的不变式中（不涉及运算）。

Fork/Join框架

发表于 2019-10-21 | 分类于 高并发 |

Fork/Join是JDK 7 提供用于并发执行任务的框架。采用分治策略。将大的任务分割成若干个小任务，再将子任务的执行结果合并后形成大任务的结果。

工作窃取算法

工作窃取算法：工作中的线程从其他任务队列中窃取任务来执行。

从上图可以看到。任务队列是一个双端的队列。线程的待执行任务都是按照FIFO策略从头部进入。而当有空闲线程时，会随机从一个任务队列的尾部窃取一个工作任务执行。线程的任务从头部进入，窃取的线程从尾部窃取任务，这样尽最大可能的减少资源冲突。

优势

相对于一般的线程池实现,fork/join框架的优势体现在对其中包含的任务的处理方式上.在一般的线程池中,如果一个线程正在执行的任务由于某些原因无法继续运行,那么该线程会处于等待状态.而在fork/join框架实现中,如果某个子问题由于等待另外一个子问题的完成而无法继续运行.那么处理该子问题的线程会主动寻找其他尚未运行的子问题来执行.这种方式减少了线程的等待时间,提高了性能。

Fork/Join

ForkJoinTask

ForkJoinTask 提供fork()来执行子任务，join()合并子任务结果的两种操作。ForkJoinTask有两个子类：

  • RecursiveAction：用于没有返回结果的任务。
  • RecursiveTask ：用于有返回结果的任务。

ForkJoinPool

ForkJoinTask需要放到ForkJoinPool中执行。任务分割出的子任务会添加到当前工作线程对应的双端队列头部。当工作线程的任务队列没有任务时，会从其他任意任务队列的尾部中窃取任务来执行。

Java 8通过发布新的Date-Time API (JSR 310)来进一步加强对日期与时间的处理。

旧版本问题

  • 非线程安全： java.util.Date是非线程安全的。
  • 设计差： Java中日期和时间的定义不一致。java.sql和java.util中都有日期类。对于java.util.Data包含日期和时间，而java.sql.Data仅仅包含日期。
  • 时区处理麻烦： 日期类并不支持国际化，没有时区支持。需要通过java.util.Calendar和java.util.TimeZone来处理。

新版本

组成说明如下表所示：

java.time.temporal 底层框架和拓展的支持
java.time.zone 时区的处理

整个time包的特征：

  • 定义了日期-时间概念的规则。包括instants、durations、dates、times、time-zones and periods。这些都是基于ISO日历系统。遵循Gregorian
  • 所有类都是不可变、线程安全的。

  • accept(T,U) 提供对数据的处理逻辑
  • andThen(BiConsumer<T,U> after) 返回一个新的BiConsumer 其accept逻辑是先执行当前accept，成功后再执行after的accept逻辑。

  • R apply(T p) 根据输入参数 处理得到返回类型数据
  • compose(Function<K,T> before) 返回一个新的Function实例，新实例的apply先执行before 如果before执行无异常 before的输出结果作为原先apply逻辑的输入，最终执行结果作为返回结果。如果before执行报错则中断执行。
  • andThen(Function<R,V> after) 返回一个新的Function实例，新实例的apply先执行原有的apply逻辑，如果原有的apply无异常，则原先apply执行结果作为after的输入参数，after执行结果作为返回结果，如果原先的apply执行报错则中断执行。
  • identity(T t) 将输入结果作为输出结果。

  • R apply(T t,U u) 根据参数t 和 u来处理获取返回类型数据
  • V andThen(Function<R,V> after) 执行apply逻辑，根据t 和 u生成结果R 再作为afterd的输入数据生成结果V 并返回。

stream

发表于 2019-07-08 | 分类于 java8 |

JDK1.8 提供了流Stream的API。将数据元素集合看作流，流在管道中传输，并且可以在管道的节点对数据进行筛选、排序，聚合等处理。

什么是流

流(Stream)是来自数据源可以进行聚合操作的元素队列。

  • 数据源：流的来源，可以是集合，数组，I/O channel等。
  • 聚合操作：如filter、map、reduce、find、match、sort等。

流操作的特点

  • 针对流的操作都会立即返回流对象本身，这样多个流的操作会串联成一个管道，流式编码风格(fluent style)，便于对操作进行延迟执行和短路。
  • 集合之前的遍历都是通过迭代器iterator或者forEach的方式，显示的从集合外部进行迭代，而流采用访问者模式进行内部迭代。

内部迭代与外部迭代的差异

方式 可控 顺序 可并行
外部迭代 ✅ ✅ ❌
内部迭代 ❌ ❌ ✅

内部迭代的顺序不一定顺序，❌仅仅表示不一定。
内部迭代的可并行通过parallelStream()方法，可以并行操作，大幅度提升性能。

如何操作流

对于流的操作，其实主要就是由生成流、中间操作和最终操作三部分组成。
对于一次的流操作，可以有多个中间操作，有且只有一次最终操作。其中中间操作也称为转换操作，即多次中间操作是在对流数据进行转换，这种转换的效率并不受数据的量级影响，转换操作反而是一种惰性求值的过程。而有切只有一次的最终操作其实是将前面所有的转换处理一次执行，这一步会受到数据量级的影响，属于及早求值。

  • 如果两个流中有一个是并行的，则新的流也是并行的。
  • 关闭新的流时，两个流也会关闭。

flatMap

flatMap与map类似，都是遍历Stream元素，并对元素做转换，不同之处是，并不是直接在Stream中用转换后的元素替换对应的元素。而是直接构造一个新的Stream。如果构造的新stream为空

样例中通过最终操作forEach来遍历Stream中的元素，而peek的执行逻辑会优先于forEach的操作。同时从输出结果可以看到，peek的Consumer逻辑处理结果并不影响原有的Stream元素。

找出stream元素排序后的最后一个元素。

min

找出stream元素排序后的第一个元素

reduce

allMatch

用于判断stream元素是否全部满足条件，如果全部满足返回true否则false。

anyMatch

用于判断stream元素中是否有一个元素满足条件，任意一个元素满足返回true否则false。

noeMatch

用于判断stream元素是否所有都不满足条件，如果都不满足返回true否则false。

findAny

用于获取含有stream元素的optional。如果stream为空，则optional为空，此操作不是确定的。并行操作时，可能会获取不同的结果，串行操作感觉是获取的第一个元素

findFirst

用于获取含有stream第一个元素的optional。如果stream为空则optional为空

limit

截取stream，如果limit保留的个数大于stream的元素个数，则返回stream全部元素，否则只保留指定个数的元素。
