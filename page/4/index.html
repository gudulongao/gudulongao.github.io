概述

zookeeper curator基本用法

发表于 2019-03-27 | 分类于 ZooKeeper |

简介

Curator是Netflix公司开源的一个Zookeeper客户端，后捐献给Apache，Curator框架在zookeeper原生API接口上进行了包装，解决了很多ZooKeeper客户端非常底层的细节开发。提供ZooKeeper各种应用场景(recipe， 比如：分布式锁服务、集群领导选举、共享计数器、缓存机制、分布式队列等)的抽象封装，实现了Fluent风格的API接口,是最好用，最流行的zookeeper的客户端。

Curator优势

  • 能够递归创建和删除节点
  • 重试机制：提供可插拔的重试机制，对可捕获的可重试异常配置重试策略，框架也提供几种标准的重试策略。
  • 连接状态监控：curator创建后会一直对zk连接进行监控，可以对状态的变化作出响应。
  • zk客户端实时管理：对zk客户端的连接进行管理，并在需要的时候重建zk客户端，保证连接的可靠性。
  • 各种场景支持：curator实现了zk的大部分场景，这些实现都遵循zk的最佳实践，并考虑了各种极端情况。

兼容性

zookeeper 3.5版本的可以使用curator4.0
zookeeper 3.4版本的可以使用curator3.X

目前我的zk版本是3.4，故我可以使用curator3.X版本。

zookeeper 实现分布式锁

发表于 2019-03-25 | 分类于 ZooKeeper |

基于上图中原理进行实现。

  • 以/lock目录为根目录，加锁的时候，先基于资源名称在/lock目录下看是否有/lock/lock_[resName]节点存在。
  • 如果/lock/lock_[resName]不存在。则创建永久节点，并且节点的值=true。当前线程成功加锁。此次加锁称之为资源锁。
  • 如果资源锁已存在，则在资源锁的目录下创建临时有序的子节点。当前线程创建的子节点序号如果是最小的，则当前线程成功加锁，此次加锁称之为临时锁。
  • 如果当前线程创建的临时子节点序列并不是最小的，则获取资源锁下的所有临时子节点，并获取当前线程序列之前的序列。加锁失败。
  • 加锁失败，需要等待锁的释放，需要判断是阻塞在资源锁还是临时锁。
  • 如果资源锁的值=true，则表明当前线程阻塞在资源锁，这个时候需要对资源锁的值变更事件加监听，阻塞当前线程直到监听的事件触发。再次尝试加锁。
  • 如果资源锁的值=false,则表明当前线程阻塞在临时锁，这个时候需要当前序列之前的临时节点删除事件加监听，阻塞当前线程直到事件触发，再次尝试加锁。
  • 释放锁时也需要判断当前线程拥有的是什么类型的锁，如果是资源锁，则将资源锁的值=false，如果是临时锁，则删除对应的临时节点即可。

zookeeper 权限管理

发表于 2019-03-21 | 分类于 ZooKeeper |

ZK的权限管理主要是依靠节点的ACL(access control list)来实现的。
节点的ACL由权限验证模式(Scheme)、权限(permission)和用户三部分组成。

Permission

  • C(create)：创建子节点
  • D(delete)：删除子节点
  • R(read)：访问节点/子节点
  • W(write)：更新节点数据
  • A(admin)：管理员

Scheme

类型 说明 写法
world 不进行权限验证 world:anyone:[permission]
ip 指定的ip才可以使用节点 ip:[ip]:[permission]
auth 指定的用户才可以使用节点 auth:user:passwd:[permission]
digest 指定的用户才可以使用节点，但是密码需要输入密文 digest:user:BASE64(SHA1(passwd)):[permission]

setAcl /path auth:lee:lee:cdrwa 与 setAcl /path digest:lee:BASE64(SHA1(password)):cdrwa是等价的，在通过 addauth digest lee:lee 后都能操作指定节点的权限。在实际情况中，digest要更为常用一些。

操作指令

  • getAcl 获取节点的acl
  • setAcl 给节点设置acl
  • addauth 进行用户认证
