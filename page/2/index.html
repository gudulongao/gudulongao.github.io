新日期API

发表于 2019-09-15 | 分类于 java8 |

Java 8通过发布新的Date-Time API (JSR 310)来进一步加强对日期与时间的处理。

旧版本问题

  • 非线程安全： java.util.Date是非线程安全的。
  • 设计差： Java中日期和时间的定义不一致。java.sql和java.util中都有日期类。对于java.util.Data包含日期和时间，而java.sql.Data仅仅包含日期。
  • 时区处理麻烦： 日期类并不支持国际化，没有时区支持。需要通过java.util.Calendar和java.util.TimeZone来处理。

新版本

组成说明如下表所示：

包名 作用
java.time 基本对象的封装
java.time.chrono 提供对不同日历系统的访问
java.time.format 格式化和解析时间日期
java.time.temporal 底层框架和拓展的支持
java.time.zone 时区的处理

整个time包的特征：

  • 定义了日期-时间概念的规则。包括instants、durations、dates、times、time-zones and periods。这些都是基于ISO日历系统。遵循Gregorian
  • 所有类都是不可变、线程安全的。

Optional是一个容器，可以很好的解决空指针问题。

JDK受Google Guava项目启发，引入了Optional类，Guava通过使用检查空值的方式来防止代码污染，它鼓励程序员写更干净的代码。Optional类已经成为Java 8类库的一部分。Optional实际上是个容器：它可以保存类型T的值，或者仅仅保存null。Optional提供很多有用的方法，这样我们就不用显式进行空值检测。

Stream的collect方法，即收集器，只要注入转换的方法，通过collect方法就可以基于stream中的元素来生成我们想要的数据结构。

辅助接口

Supplier

Supplier接口主要是提供方法用于获取指定类型的对象。

BiConsumer

  • accept(T,U) 提供对数据的处理逻辑
  • andThen(BiConsumer<T,U> after) 返回一个新的BiConsumer 其accept逻辑是先执行当前accept，成功后再执行after的accept逻辑。

样例

通过样例验证了 andThen是生成了一个新的BiConsumer实例，新的accept逻辑是先执行left的accept之后再执行after的accept。同时left的accept不能报错。

Function

  • R apply(T p) 根据输入参数 处理得到返回类型数据
  • compose(Function<K,T> before) 返回一个新的Function实例，新实例的apply先执行before 如果before执行无异常 before的输出结果作为原先apply逻辑的输入，最终执行结果作为返回结果。如果before执行报错则中断执行。
  • andThen(Function<R,V> after) 返回一个新的Function实例，新实例的apply先执行原有的apply逻辑，如果原有的apply无异常，则原先apply执行结果作为after的输入参数，after执行结果作为返回结果，如果原先的apply执行报错则中断执行。
  • identity(T t) 将输入结果作为输出结果。

样例


通过样例验证了compose方法和andThen的逻辑。

BiFunction

  • R apply(T t,U u) 根据参数t 和 u来处理获取返回类型数据
  • V andThen(Function<R,V> after) 执行apply逻辑，根据t 和 u生成结果R 再作为afterd的输入数据生成结果V 并返回。

样例

BinaryOperator

  • minBy(Comparator c) 构建一个新的BinaryOperator实例，其apply逻辑是通过比较器获得两个输入参数中比较后小的值。
  • maxBy(Comparator c) 与上面的类似，只是获取输入参数重比较大的值。

BinaryOperator是继承了函数接口BiFunction的，即根据两个输入参数来获取结果数据，由于要对输入的两个参数进行比较，所以参数类型要一致，故BiFunction的三个参数类型都是一样的

stream

发表于 2019-07-08 | 分类于 java8 |

JDK1.8 提供了流Stream的API。将数据元素集合看作流，流在管道中传输，并且可以在管道的节点对数据进行筛选、排序，聚合等处理。

什么是流

流(Stream)是来自数据源可以进行聚合操作的元素队列。

  • 数据源：流的来源，可以是集合，数组，I/O channel等。
  • 聚合操作：如filter、map、reduce、find、match、sort等。

流操作的特点

  • 针对流的操作都会立即返回流对象本身，这样多个流的操作会串联成一个管道，流式编码风格(fluent style)，便于对操作进行延迟执行和短路。
  • 集合之前的遍历都是通过迭代器iterator或者forEach的方式，显示的从集合外部进行迭代，而流采用访问者模式进行内部迭代。

内部迭代与外部迭代的差异

方式 可控 顺序 可并行
外部迭代 ✅ ✅ ❌
内部迭代 ❌ ❌ ✅

内部迭代的顺序不一定顺序，❌仅仅表示不一定。
内部迭代的可并行通过parallelStream()方法，可以并行操作，大幅度提升性能。

如何操作流

对于流的操作，其实主要就是由生成流、中间操作和最终操作三部分组成。
对于一次的流操作，可以有多个中间操作，有且只有一次最终操作。其中中间操作也称为转换操作，即多次中间操作是在对流数据进行转换，这种转换的效率并不受数据的量级影响，转换操作反而是一种惰性求值的过程。而有切只有一次的最终操作其实是将前面所有的转换处理一次执行，这一步会受到数据量级的影响，属于及早求值。

生成流

  • Stream接口的静态工厂方法。
  • Collection接口和数组的默认方法，可以转换成Stream对象。
  • 其他

中间操作

中间操作主要是针对Stream进行一些转换和限制操作。属于延迟计算。

concat

将两个流连接成为一个流。

  • 如果两个流有都是有序的，则新的流也是有序的。
  • 如果两个流中有一个是并行的，则新的流也是并行的。
  • 关闭新的流时，两个流也会关闭。

distinct

将Stream中重复的元素进行过滤.

filter

对Stream根据指定的条件进行过滤。

map

将Stream中指定的元素进行转换，返回的Stream中只保留转换后的元素。

Stream中已默认包含mapToDouble、mapToInt和mapToLong三个方法，即Stream默认支持将元素转换成Double、int和Long三种类型。

flatMap

flatMap与map类似，都是遍历Stream元素，并对元素做转换，不同之处是，并不是直接在Stream中用转换后的元素替换对应的元素。而是直接构造一个新的Stream。如果构造的新stream为空

peek

peek方法的参数是一个Consumer函数接口实例。其逻辑是在对Stream中的每一个元素在被消费的时候，都会事先执行Consumer逻辑。同时Consumer逻辑并不影响Stream中的元素。

样例中通过最终操作forEach来遍历Stream中的元素，而peek的执行逻辑会优先于forEach的操作。同时从输出结果可以看到，peek的Consumer逻辑处理结果并不影响原有的Stream元素。

skip

sikp方法的意义在于过滤Stream指定个数的元素。如果Stream的元素个数大于指定要过滤的个数。则返回一个length-n 个元素的新stream，否则返回一个空的stream。

默认方法

默认方法：在接口中用default修饰的方法，接口中可以直接提供该方法的实现逻辑。即实现类可以不用实现该方法，直接使用接口中提供的实现方法。

默认方法的作用是解决接口的修改与现有的实现不兼容的场景。当接口发生变化时，其实现类很多，这个时候可以提供默认方法，实现类可以不用关注接口的变更。

子接口继承多个接口有相同的默认方法，或者实现类实现多个接口有相同的默认方法时，子接口和实现类必须实现相同的默认方法，否则会有编译错误。

  • 可以通过接口class.super.method来指定调用的默认方法。([className].super.[methodName])
  • 子接口或者实现类可以定义自己的默认方法对上层的默认方法进行覆盖。

不能对Object的方法提供默认方法即不能重写Object中的方法，可以重载。

接口静态方法。

  • 1.8版本接口可以声明静态方法。
  • 子接口或者实现类不会继承静态方法
  • static与default不能同时使用

函数式接口

发表于 2019-07-01 | 分类于 java8 |

函数式接口(Functional interface)：有且仅有一个抽象方法（可以有多个非抽象方法）。

  • 函数式接口可以由lambda表达式实现(1.8以前都是通过匿名内部类来实现)
  • 使用注解@FunctionalInterface 来启动函数式接口编译检查，同时会更新JavaDoc来表明当前接口是函数式接口

1.8在java.util.function包下增加了如下函数式接口。

@注解FunctionalInterface只是帮助编译器做检查，不会有实际的含义。即当声明为函数式接口，只能有一个抽象方法。可以有多个实现方法。可以有多个默认方法(default修饰的方法)。
