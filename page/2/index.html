网络分层模型

发表于 2018-09-13 | 分类于 网络 |

OSI七层模型

OSI(Open System Internet)开放式系统互联网。

应用层

面向实际的应用程序，向应用程序提供网路服务。
常用的网络层协议有：

  • HTTP 超文本传输协议
  • FTP 文件传输协议
  • TFTP 简单文件传输协议
  • SMTP 简单邮件传输协议
  • NTP 网络时间协议
  • POP3 邮件协议第三版
  • IMAP4 网络信息访问协议第四版

两国之间可以通过书信，大使互访，邮件等多种方式来沟通互信，每种方式都有着各自的规矩。目前采用书信来举例。
故应用层中提供了多种网络服务的协议，每种协议都有着不同的处理方式。

表示层（编码，加密，压缩）

当两国文字不同语言不通时，这种互通需要翻译将本国语言书写的内容翻译成标准语言，方便对方能正常解读。同时为了安全，还会进行加密处理。

故应用层的数据编码和转换（加/解密和数据压缩），确保应用层的数据能够被别的系统应用层能够正常识别。

会话层

外交部拥有对方国家邮箱的地址，当外交部收到来自翻译后的国书后，外交部将建立本次会话的记录。外交部将国书中增加对方国家的地址信息。准备将国书进行投递。当收到对方的回信，则此次会话结束。

故会话层是负责建立，管理表示层之间的通信会话。

传输层（端到端）

邮递员负责邮件的收发工作。

传输层负责了端到端的链接（端口号），为会话层提供了可靠透明的数据传输服务。包括差错处理和流量控制。
包含协议：TCP/UDP。

网络层（IP协议层）

负责传输邮件的是复杂的快递网络。覆盖各地的集散中心，从端到端的传递需要经过集散中心。

负责通过IP寻址来建立端到端的链接，选择合适的路由和交换节点，将数据正确无误的传输到目的端。

数据链路成

将数据组合成帧，以MAC地址来访问，并提供差错检测。

物理层

以物理介质来传输比特流。

对象的创建存储以及访问

发表于 2018-09-12 | 分类于 jvm |

对象的创建

当遇到new指令时，jvm其实就在进行对象的创建。对象的创建有以下几个步骤：

类加载检查。

在运行时常量池中寻找类的符号引用，并检查符号引用对应的类是否被加载、解析和初始化。如果没有则进行类加载。

分配内存。

对象所需要的内存空间大小在类加载之后就能完全确定。所以首先需要类加载的过程。对象在堆中分配内存的方式主要有两种：指针碰撞和空闲列表。

指针碰撞

堆空间很规整，即使用的空间存放在内存中的一边，空闲的在另一边，使用和未使用的空间分界为指示器，分配内存只需要将指示器向空闲区域挪一段与新对象大小相当的距离即可。

空闲列表

堆空间不规则，已使用和未使用的内存区域零散分布。jvm维护着一个空闲区域的列表，记录可用的内存空间，为对象分配内存的时候只需要选择一个足够大的空间分配给新的对象并更新列表。

线程安全的分配内存

  • 分配内存的同步处理：jvm采用CAS策略以及失败重试的方式来保证分配内存操作的原子性。
  • 分配内存根据线程来划分区域：TLAB(Thread Local Allocation Buffer)，堆中每个线程都会预先分配一块内存，每个线程需要分配内存时都是在自己的TLAB中分配，只有自己的TLAB分配完毕才会同步锁定。

内存初始化（零值）

将分配到的内存空间初始化为零值。

保证对象的实例字段在java代码中可以不赋初始值就直接使用，程序能访问到这些字段的数据类型所对应的零值。

初始化对象头

对对象进行必要的设置，例如这个对象是哪个类的实例，如何才能找到累的元数据信息、对象的哈希码、对象的GC分代年龄等。

初始化

把对象按照程序员的意愿进行初始化。

个人感觉这一步就是调用初始化方法。

对象的内存布局

对象在内存中存储的布局可以分为3块区域：对象头，实例数据和对其填充。

对象头

对象头包含两部分：对象运行时数据和类型指针。
如果对象是数组数据，对象头还必须有一块用于记录数组长度的数据。

对象运行时数据（Mark Word）

包含哈希码、GC分代年龄、锁状态标志、线程持有的锁、偏向线程ID、偏向时间戳等信息。
数据长度在32位和64位虚拟机中分别为32位和64位。

类型指针

指向类 元数据的指针，虚拟机通过这个指针来确定对象是哪个类的实例。
并不是所有的虚拟机都必须在对象数据上保留类型指针。

实例数据

对象真正存储信息的部分。即实例对象中所有的字段内容。包含从基类中继承的字段。

对其填充

非必须存在。当对象的实例数据部分数据没有对齐时需要对其填充来补全占位。

对象的访问

句柄池访问

如图所示，堆空间会存在句柄池和实例池，句柄池中的句柄包含实例对象的地址和对象类型的指针。而栈帧中的局部变量表中的reference指向句柄池中的句柄。
使用句柄池的好处是reference中存储的是稳定的句柄地址，当GC垃圾回收导致对象的地址变动的时候，只需要变更句柄池中实例对象指针的地址，无需修改reference。

直接指针访问（Sun HotSpot采用）

如图所示，栈帧中局部变量表的reference直接指向实例对象，实例对象对象头的类型指针指向类型数据。
直接指针的好处是因为少一次指针定位的时间开销，故对象访问的速度更快。

JVM内存分布

发表于 2018-09-10 | 分类于 jvm |

程序计数器

当前线程执行字节码的行号指示器，说白了就是存储当前线程执行指令到哪个位置了。

  • 线程私有（每个线程都有自己的程序计数器，互补影响，独立存储）。
  • 执行java方法时，存储执行字节码行号的执行器，执行native方法时为空。
  • 不会抛oom异常。

虚拟机栈

描述执行java方法时，每个方法执行时都会创建一个栈帧用于存储局部变量表，方法出入口等信息。每个方法的调用直至完成执行的过程，都是栈帧入栈出栈的过程。

个人感觉虚拟机栈，就是一个栈，在每次调用方法的时候，会创建栈帧来存储线程的局部变量表，方法的出入口地址等信息，方法调用就将栈帧压入虚拟机栈，方法执行成功则弹栈。

局部变量表

局部变量表存放了编译期间基本类型（boolean,byte,char,short,int,float,long,double）和对象引用类型等变量所需的内存空间。当进入一个方法，这个方法需要在栈帧中分配多大的局部变量空间是完全确定的，在方法运行期间不会改变局部变量表的大小。

个人感觉局部变量表就是一个专门用于存储在这个方法作用于所有的基本类型变量和引用类型变量的内存区域，随着进入方法的时候创建，随着离开方法时销毁。

  • 线程私有。
  • 当线程请求的栈深大于虚拟机允许的栈深，抛出StackOverFlow异常。
  • 当虚拟机栈动态拓展时无法申请到足够的内存时，抛出OutOfMemory异常。

本地方法栈

类比虚拟机栈，只不过本地方法栈是线程调用本地方法时，所涉及到的内存区域。

  • 会抛出StackOverFlow和OutOfMemory异常。

堆

所有对象实例以及数组都要在堆分配。

  • 所有线程共享。
  • 垃圾回收的主要区域。
  • 物理无需连续，逻辑上连续的内存空间。
  • 如果没有空间给对象分配内存，同时堆也无法继续拓展，则会抛出OutOfMemory异常。

方法区（非堆 Non-Heap）

用于存储已被虚拟机加载的类信息、常量、静态变量等数据。

  • 线程共享。
  • 当没有空间满足内存分配时，抛出OutOfMemory异常。

运行时常量池

方法区的一部分。
Class文件中除了有类的版本、字段、方法、接口等描述信息外，还有常量池，用于存放编译期间生产的各种字面量和符号饮用，这部分内容在类加载后进入方法区的运行时常量池存放。
java语言并不要求常量一定只有编译期间才能产生，运行期间也可能将新的常量放入常量池。例如String类的intern方法。

感觉运行时常量池就是用来存储类的相关信息，包括类的常量，静态变量等信息。同时String类的intern也会将字符串常量添加到运行时常量池中。

字符串的使用

new String（） 的存储

new String(“abc”)，会直接在堆空间中分配空间来存储字符串abc。

通过new String是直接实例化String对象，实例化对象都是在堆空间中分配内存，故例子中str1与str2存储着两块不同的堆空间地址。

字符串拼接和常量拼接

字符串的拼接以及常量的拼接，java会在编译期间就会做拼接，即在编译期间，str9=”str8”,str10=”str8”。当前代码是在一个方法中，故属于方法的局部变量，所以此处的三个变量位于虚拟机栈的局部变量表中。

变量与字符串拼接

当字符串与变量拼接的时候，jvm底层会通过StringBuilder.append在堆上构建新的对象。字符串对象是不可变的，所以对于这种字符串与变量的拼接，实际创建一个新的字符串对象，内容是拼接的字符串。


通过append的源码可以看到，StringBuilder其实底层是一个数组，每次append的时候，把字符串的每个字符都插入到数组里。在toString方法中构建一个新的字符串对象。

String.intern方法

jvm为了使得同样的字符串能够复用而不再分配内存，减少内存的消耗。

1.6版本

在1.6版本中，定义字符串（String str1 = “abc”）时，会先查询运行时常量池，如果有一致内容，则直接返回运行时常量池的引用，即str1直接指向运行常量池中字符串”abc”。如果没有，则直接在运行时常量池中自动添加一个字符串”abc”，然后返回字符串”abc”的引用。在1.6版本中运行时常量池中还是会存储字符串数据的。

1.7版本


在1.7版本中，定义字符串变量的时候，一样会先在运行时常量池中查找是否有内容一致的字符串，有则返回运行时常量池的引用。如果没有，则会在堆中创建字符串对象，再在运行时常量池中引用堆中的字符串对象。

总结

  • 不论1.6还是1.7，字符串变量都是指向运行时常量区。只不过1.6中运行时常量池直接存储字符串内容。而1.7版运行时常量池存储的事堆中字符串对象的引用。
  • 相同内容的字符串变量其实都指向运行时常量区中同一块内存空间。

线程池

发表于 2018-09-10 | 分类于 高并发 |

为什么要使用线程池

  • 线程的创建和销毁都需要耗费时间
  • 线程也是需要瓜分内存空间，不加节制的增加线程会造成内存不足（oom异常）
  • 大量的线程造成的内存耗费，导致垃圾回收的时间加长（gc停顿）

JDK提供的线程池

Executor框架提供了几种线程池，可以通过Executors工厂类来获取。

固定线程数线程池（newFiexdThreadPool）

固定线程数线程池，其线程数量在初始化时已固定，不会发生变化，提交任务时，如果有空闲线程时，则立即执行。如果没有，任务会进入一个任务队列，从源码中可以看到其任务队列为LinkedBlockingQueue。

单个线程线程池(newSingleThreadExecutor)

单一线程池，其线程数只有一个，当提交任务多于1个时，任务会进入任务队列，待线程空闲时按照任务队列中先进先出的顺序来执行。通过源码可以看到，单个线程池与固定线程数线程池的实现逻辑类似，都是使用LinkedBlockingQueue来作为任务队列的。

不定线程数线程池(newCachedThreadPool)

不定线程数线程池，其线程数量是动态变化的，提交任务时，如果有空闲线程，则立即执行，如果没有，则会创建新的线程处理任务。当所有执行完毕任务后，会返回到线程池中等待复用。从源码冲可以看到，其使用SynchronsQueue作为任务队列。

单个定时任务线程池（newSingleThreadScheduledExecutor）

线程数为1的可定时执行任务的线程池。

定时任务线程池（newScheduledThreadPool）

可以指定线程数的可执行定时任务的线程池。

执行定时任务

对于定时任务的线程池，其包含两个方法，用来运行定时任务。

  • scheduleAtFixedRate 以上一个任务的开始时间为起点，等待period时长之后再调用下一个任务。即period的时长是包括上一个任务的运行时间的，如果上一个任务运行时长>period时长，那么上一个任务执行完毕后会立即执行下一个任务。
  • scheduleWithFixedDelay 以上一个任务的结束时间为起点，等待delay时长以后再调用下一个任务。即下一个任务等待的时长=上一个任务运行的时长+delay时长。

线程池的内部实现

从jdk中提供的几种线程池的源码中可以看到，其内部都是用了ThreadPoolExecutor。

任务队列

创建线程池的时候，需要指定线程池中的人物队列，从ThreadPoolExecutor的构造函数中可以看到，任务队列都是实现了接口BlockingQueue的队列对象。常用的人物队列有有SynchronousQueue，ArrayBlockingQueue，LinkedBlockingQueue，PriorityBlockQueue。

零界队列（SynchronousQueue）

SynchronousQueue 是一个没有容量的队列。每一次插入操作都必须等待一个删除操作。反之，每一个删除操作都等待对应的插入操作。即一旦有元素进入队列（生产者插入）会被立即删除（消费者删除），更像是生产者利用该队列快速的将元素传递给消费者。
newCachedThreadPool的线程池就使用了SynchronousQueue来作为任务队列，当有空闲线程时立即执行任务，当没有空闲线程的时候，任务会插入到SynchronousQueue中，由于SynchronousQueue的特性，当插入的时候就必须要有对应的删除，即需要立即创建新的线程从任务队列中获取任务。如果当前线程数已到达最大线程数。会触发拒绝策略，故使用SynchronousQueue的线程池其核心线程数要很大，否则很容易触发拒绝策略。

有界队列（ArrayBlockingQueue）

ArrayBlockingQueue 是一个有界队列。构造时需要指定初始容量。
当有任务时，线程池处理线程的逻辑如下：

  • 线程数<coreSize，创建线程，立即执行任务。
  • 线程数>coreSize，将任务添加到任务队列，等有空闲线程时再从线程中获取任务执行。
  • 任务队列已满，线程数<maxSize，创建新线程执行任务。
  • 任务队列已满，线程数=maxSize，触发拒绝策略。

当时看到线程池基于有界队列任务的线程维护策略中对紧急状态时有些疑惑。故写了样例。样例中通过有界队列ArrayBlockingQueue来作为线程池的任务队列，线程池中提交了两种任务，一种长时间运行一种段时间运行。主要是为了测试当任务来临时，任务队列已满，线程数<maxSize时会创建新的线程来执行任务，当最终任务执行完毕后，线程池中还保留的线程数是coreSize还是>coreSize，通过查看输出可以看到，当紧急任务来临时，线程池会拓展线程数到maxSize,当任务执行完毕后，线程池并不会将>coreSize的线程给销毁，会保留。

无界队列（LinkedBlockingQueue）

  • 线程数<coreSize，新建线程，立即执行任务。
  • 线程数>coreSize，将任务添加到对了，等待有空闲线程时执行任务。由于任务队列是无界队列，可以不断的增加队列长度，故直到系统资源耗尽，任务会一直添加到任务队列，也不会触发拒绝策略。

优先级队列（PriorityBlockQueue）

线程池会根据任务的优先级来优先选择优先级高的任务优先执行。

拒绝策略

对于有界队列或者零界队列时，当没有空闲线程，线程数到达maxSize则会触发拒绝策略。jdk提供的4种拒绝策略：

  • AbortPolicy 直接抛出异常。
  • CallerRunsPolicy 如果线程池没有停止，则直接在调用该任务的线程中执行。
  • DiscardOledestPolicy 丢弃最老的任务，由于除了优先级队列，有界和无界队列都是按照FIFO先进先出策略来处理任务，故最老的任务就是即将要被执行的任务。
  • DiscardPolicy 直接丢弃当前任务。

其他三个策略都好理解，对于CallerRunsPolicy，我的感觉就是当线程数到达maxSize的时候，对于新提交的任务，会直接在调用线程池提交任务的线程中来执行该任务，从而使得提交任务的线程被迫暂停原由的工作转而执行任务。

重入锁(ReentrantLock)

实现接口Lock。显示进行加锁，释放锁的操作使得并发处理时更加的灵活。

同步代码块（lock/unlock）

如图是使用重入锁来进行并发操作的样例，可以看到两个线程都共享了重入锁lock，在同步代码块的开始需要手工显示的进行加锁。在同步代码块的最后，需要手工显示的释放锁。
重入锁名称的缘由是指在同一个线程内，可以多次进行加锁，同时，最终也要对等的多次进行释放锁。

响应中断(lockwithinterruptibly)

重入锁相比synchronized，在线程处理临界资源的时候，除了获取锁继续执行和阻塞之外，使用重入锁还有个优势就是可以支持中断。即在未获取到临界资源导致线程阻塞的时候，可以通过响应中断来停止阻塞的状态。这种响应中断的方式对于解决死锁有很大的帮助。

上面的样例就是营造一个死锁的场景，FirstThread运行后会获取readlock，成功后又想获取writelock。而AfterThread运行后会先获取writelock，成功后想获取readlock。这个时候两个线程就发生了死锁。两个线程都无法完成自己的工作。这个时候对FirstThread进行了中断，在FirstThread处理中断异常的时候，释放了所拥有的所有资源，而AfterThread便成功获取了readlock而完成工作。
重入锁的lockWithInterruptibly方法会有可能抛出InterruptedException异常，所以可以在catch里能够专门处理中断时的场景。

限时等待(trylock，解决死锁)

为了避免死锁，除了支持中断的方式之外，重入锁还提供了这种限时等待的方式来避免发生死锁。

根之前的样例类似，每个线程都是在不断的循环直到完成工作，FirstThread会先获取lock1的锁，再获取lock2的锁。如果没有能成功，则在finally里释放锁。AfterThread的内容一致只是获取锁的顺序不一样，先获取lock2的锁再获取lock1的锁。如果都不成功则释放所有的锁。因为两个线程获取到锁以后的等待时长是不一致的，所以总会有一个线程释放所有资源之后另个线程能够拿到所有所需的资源继续执行。这样就不会导致死锁发生。
重入锁提供trylock方法来解决死锁问题，无参的trylock表示立即获取锁，如果成功则返回true，否则返回false，不会阻塞等待，有参的trylock有有两个参数，第一个是时长，第二个是时长的单位，指如果没有获取到锁会阻塞等待的时长，不会像synchronized那样一直阻塞到获取到锁。

公平锁（解决饥饿）

线程运行的时候因为线程的优先级会导致优先级高的线程总会有更高的概率获取临界资源。就会导致优先级的线程总示无法获取临界区资源从而产生饥饿现象。
重入锁的构造可以注入是否采用公平锁，如果true则表明所有的线程在获取锁的时候按照申请的时间来排序，即先申请获取锁的线程先获取。

缺陷

公平锁需要额外的维护一个有序队列来保证线程申请锁的顺序。实现成本高，性能不如默认的非公平锁。

等待和继续（Condition）

Object对象的实例方法wait/notify与synchronized结合，可以控制线程等待和继续执行。重入锁可以结合Condition的await/signal达到一样的效果。

  • Condition是从重入锁而来。
  • 与Object的wait和notify一样，使用condition的await/signal的时候，也必须先获取重入锁。

与synchronized的区别

  • synchronized不支持中断，即使用synchronzied处理并发的时候，线程只有两种状态：要么拿到锁成功执行，要么等待阻塞直到获取到锁。而对于重入锁，可以通过lockwithinterruptibly来支持中断响应，也可以通过trylock来限时等待锁。
  • synchronized容易产生死锁，即获取不到锁线程会一直阻塞也不释放资源。而重入锁在lockwithinterruptibly方法支持中断响应，即在获取不到锁阻塞的时间，会抛出InterruptException异常，我们可以在异常中释放资源，这样避免死锁的发生，甚至可以通过trylock的方式来获取锁，限时等待锁的时长甚至根本不等待。
  • synchronized结合Object对象的实例方法wait/notify来让线程等待/恢复执行。重入锁结合Condition的await/signal来让线程等待/恢复执行。
  • synchronzied是非公平锁，重入锁默认是非公平锁，支持公平锁。

循环栅栏（CyclicBarrier）

先上样例代码

  • 循环栅栏的构造函数有两个参数，一个是指定需要栅栏拦截停止的线程数，另一个是当拦截停止的线程数足够的时候，执行的栅栏任务。
  • 多线程任务需要共享同一个栅栏对象，每当多线程执行到栅栏的await方法时，就会进入栅栏的等待线程队列中，同时栅栏的计数器会减1。这样多线程任务会暂停，直到暂停的多线程数量符合预期之后，栅栏会执行栅栏任务。
  • 当栅栏任务执行完毕之后，栅栏会将其等待线程队列中的所有线程都恢复运行，同时计数器的计数也会重置到初始值，方便下次多线程任务进行await的时候时候。

书中的例子很生动，多线程任务就等同于点兵，栅栏任务即点兵符合点兵的数目之后执行的任务。比方说某项任务，需要集齐10个士兵上车，人员整齐之后开往目的地，10个士兵再下车，都下车之后，布置任务。10个士兵就是多线程任务，10个士兵都并发的上车，上车之后执行栅栏的await方法，暂停休息等待10个士兵都上车，即线程都await之后，人齐了，等待栅栏任务即开往目的地，等到达目的地之后10个任务重新被激活，开始并发的下车，下车之后再次调用栅栏的await方法，暂停休息等待所有士兵都下车即10个线程都await之后，等待栅栏任务即分配任务。

注意

栅栏的计数器会导致只有当拦截的线程数达到预定的数目之后才会执行栅栏任务，也只有执行栅栏任务之后暂停的多线程任务才会被重新激活继续运行，如果多线程任务数量小于栅栏预定的拦截线程数目，那么整个流程都会卡在这里。如果多线程任务的数量大于栅栏预定拦截的线程数，则栅栏计数器的功能就整个失效了，整个流程拦截暂停多线程任务的效果就呈现一种混乱的状态。栅栏预定拦截的多线程任务数量一定要与多线程任务数量一致。

阻塞工具(LockSupport)

线程阻塞的工具类，其提供静态的方法park来阻塞线程，提供静态的方法unpark来唤醒线程。

LockSupport有以下几个优点：

  • LockSupport基于信号量为1的机制，其为每一个线程都准备了一个令牌。执行park方法的时候，如果有令牌可用，则不会阻塞，如果令牌不可用才会阻塞，只有unpark方法会使得令牌可用。即如果先执行了unpark方法，也不会使得线程永久的挂起，因为只要执行过unpark方法，就会有令牌可用，这个时候再执行park方法并不会阻塞。
  • 不会抛出InterrupteException异常，但是LocSupport的park方法是支持中断响应的，即可以在调用park方法后，通过线程的中断标志来判断是否有中断，如果有则处理中断场景。

与Suspend/resume的区别

  • suspend/resume方法是Thread的实例方法，现已被标示为废弃方法。park和unpark方法为LockSupport的静态方法。
  • suspend无法处理中断。而park方法可以通过线程的中断标志来处理中断。
  • 一旦resume执行在suspend之前，suspend后面没有resume的话，线程会被永久挂起。而LockSupport则不会，基于信号量机制，哪怕unpark优先于park执行，park遍不会阻塞挂起，会立即返回。

高并发并行基础概念

发表于 2018-09-09 | 分类于 高并发 |

进程/线程

  • 程序：指令和数据的描述（指令+数据）
  • 进程：操作系统进行系统资源分配的基本单位。进程是程序的实体（程序运行起来就是一个进程）。
  • 线程：程序最小的执行单位，即进程中可以包含多个线程，进程是线程的容器。
    线程的切换以及调度的成本比多进程小。

线程的状态

在Thread的源码中有定义线程状态的枚举。可以看到线程有以下几个状态：

  • NEW：刚刚创建线程。等待调用strat方法开始运行。
  • RUNNABLE：线程正在运行。
  • BLOCKED：当遇到同步代码块会阻塞，线程会暂停执行，直到获取锁。
  • WAITING：无时间限制的等待。（wait等待notify，join等待目标线程执行结束）
  • TIMED_WAITING：有时常的等待。
  • TREMINATED：终止。

终止

终止线程的时候可以使用stop方法，但不建议使用，因为stop停止线程会让线程暴力结束，这样如果线程没有完成当前工作，比如写文件，还没有写完就被终止，这时候文件是个半成品，会造成后续的数据错误。所以对于线程如果需要终止，可以在线程中增加一个标记，通过标记来判断是否还需要继续运行。外部通过修改这个标记的值来手工达到终止线程运行的效果。

wait/notify/notifyall

Object对象中都包含wait和notify的方法。注意这一对方法是在Object对象中的。
使得当前线程进入该对象的等待队列中，直到该对象调用notify方法才会从等待队列中出来。但线程是否能够继续从等待的位置继续运行，还要看线程是否能重新获取到锁。只有重新获取到锁以后才能继续执行。

从样例代码中可以看到，两个线程共享了对象lock，每次在wait操作和notify操作的时候都首先需要获取到lock的锁，wait/notify的操作都必须保证在同步代码块中。同时也可以想到当线程1中lock执行wait之后，线程1就进入了lock对象的等待线程队列中了，这个时候线程1也就释放了lock的锁。这样线程2才有机会获取到lock的锁，从而通过lock调用notify来让暂停的线程1从lock的等待线程队列中出来。这个时候线程1还需要再次竞争lock的锁才能从等待的位置继续执行。

suspend/resume

Thread提供实例方法suspend来暂停当前线程，并不会释放锁，知道当前线程调用resume才会重新激活继续执行。
这一对方法目前已经废弃

缺陷

  • suspend之后，不会释放锁
  • 一旦resume先于suspend执行，即suspend之后再也没有resume操作，当前线程会一直挂起。并且不会释放其占有的资源，从而影响别的线程。

与Sleep的区别

  • wait操作必须在同步代码块中，同时会释放锁，而且wait方法是Object的实例方法。
  • sleep只是让线程暂停一定时长，是Thread的静态方法，并不会释放资源（躺下睡赖着不走）。

join

Thread的实例方法，会阻塞当前线程，直到目标线程运行完毕。其实质等同于当前线程进入目前线程的等待线程队列（wait操作），直到目前线程执行完毕后当前线程再继续。

yeild

Thread的静态方法，当前线程主动让出CPU，这样多线程会重新开始竞争CPU，所以当线程调用yeild之后，还是有可能又是其获取到CPU继续执行。

对于某些线程的工作并不重要，优先级不高，害怕其占用太多CPU资源，可以时常通过yeild方法，给予其他线程更多的工作机会

守护线程

在start方法之前通过setDaemon(true)来设置线程为守护线程。当所有的用户线程都结束，守护线程才会结束，jvm才会停止

线程优先级

通过实例方法setPriority来设置优先级。java中通过1到10来表示线程的优先级，数值越大的优先级越高。优先级越高越容易赢得资源的竞争。

volatile

多线程会共享JVM的堆空间，对于共享资源都在JVM的主堆空间中，而多线程之间的共享资源，都会在线程自己的工作空间中有一份拷贝副本。volatile关键词的作用是对于多线程不再工作空间拷贝主堆空间的共享资源，而是直接使用主堆中的共享资源，这样解决了多线程并发时共享资源的可见性。

根据编译器优化原则，如果不使用volatile申明变量，那么这个变量被修改后，其他线程可能并不会被通知到，甚至在别的线程中，看到变量的顺序都会是反的。但一旦使用volatile，虚拟机就会特别小心地处理这种情况。

感觉volatile关键词主要保证了共享资源在多线程中的可见性，对有序性也有所帮助。

synchronzied

指定对象加锁：进入同步代码块钱要活的给定对象的锁。

高并发基础概念

发表于 2018-09-07 | 分类于 高并发 |

同步/异步

  • 同步：调用方必须阻塞等待方法的返回后方可继续执行后续操作。
  • 异步：调用方调用后无需等待方法的返回，方法会在另一个线程去执行，整个异步执行的过程是不会影响调用法的继续工作。

并发（Concurrency)/并行（Parallelism）

如图所示：

  • 并发：多个任务交替执行。逻辑上的同时执行。
  • 并行：多个任务同时执行。物理上的同时执行。

临界区

可被多个线程共享的资源。但是临界区的资源每次只能被一个线程独占。一旦临界区资源已被占用，其余线程只能等待。

阻塞/非阻塞

描述多线程的一种并发级别。

  • 阻塞：有线程进入临界区资源，其余的线程想要进入临界区，只能等待，等待会导致线程挂起。这种会有线程挂起的并发场景就是阻塞。
  • 非阻塞：即多线程并发时不会有线程导致其他线程挂起。

死锁/饥饿/活锁

  • 死锁：每个线程都拥有别的线程继续执行的资源而有无法释放已拥有的资源，导致线程之间互相等待。
  • 饥饿：时有的线程总是无法获取临界区资源，导致其无法继续执行。
  • 活锁：线程间保持‘谦让’,即每个线程都主动释放临界资源，而导致资源在线程间来回跳动，没有一个线程可以同时拿到所有的资源继续执行。

并发级别

阻塞

  • 阻塞要求只有一个线程能够进入临界区

线程在执行前，都需要先对临界区进行加锁，而其他线程由于临界区加锁而无法继续执行导致挂起。我们常用的synchronized和ReetranLock都是阻塞锁。

无饥饿

  • 无饥饿要求每个线程都能有机会进入临界区

线程之间的优先级会导致并发的时候产生饥饿问题，而无饥饿的并发级别是指使用公平锁，即所有的线程都是平等的没有优先级的。

无障碍

  • 无障碍要求多线程并发时至少有一个线程能成功执行（走出临界区）。
  • 无障碍的并发级别可行的方案就是采用CAS策略。

每个线程都可以访问临界区资源，都可以做修改，但是修改提交时有可能因为临界区资源已被修改导致的冲突而回滚。即线程提交修改时如果没有冲突则成功提交否则需要回滚修改，直到提交成功才可以走出临界区。
无障碍是一种乐观锁，这种乐观锁的极端情况就是可能没有一个线程能够成功提交修改，都在不停的回滚。

无锁

  • 要求至少一个线程能在有限步内走出临界区
  • 其余线程可以饥饿。

一般无锁的调用会采用一个死循环，不断尝试提交修改，如果总是不能成功提交，则产生饥饿，如果成功提交，则满足在有限步内走出临界区。

无等待

  • 要求每个线程都能在有限步内走出临界区。

常用的方式是RCU(Read Copy Update)，即对读操作的线程不加锁，这样所有的读操作线程不会阻塞也不会冲突，而对于写线程在修改数据的时候，先获取数据的副本，修改数据的副本，等待合适的时机将副本数据回写同步到数据。这样写线程也没有阻塞和冲突也都能在有限步内完成工作走出临界区。

高并发操作的线程要求


如图所示，jvm中多线程的内存模型。在jvm中多线程都是共享堆空间。每一个线程都有自己的内存空间。每一个线程内存空间有共享资源的副本。
个人理解：基于RCU原理，线程每次对共享资源的修改其实都是在修改线程内共享资源的副本，为了使得共享资源能够在多线程中共享，需要考虑能够让线程内的副本的修改同步到主内存的共享资源，其余资源也要能知道共享资源发生变化，这就是需要考虑线程的可见性。

原子性

共享资源在被多线程并发操作的时候，每一个线程的操作都必须是原子的，共享资源的结果不会在一个线程做操作的时候因为收到别的线程而导致错误。

比方说共享的对象是Long型的数据，多线程对其累加修改数据，在32位的系统中，对Long型的数据修改不具备原子性，有可能一个线程只修改这个64位数据的其中32位，而另一个线程则读取了修改了一半的数据，进行累加，这个结果势必就是错误的。这种错误的数据就是因为两个线程对数据的操作没有达到原子性。

可见性

多线程并发时，一旦有线程修改了共享数据，其余线程能够立刻知道修改。

  • 可以通过volatile来修饰共享数据，这样保证多线程之前数据的可见性。

有序性

jvm因为效率的问题，会发生指令重排，即实际执行的指令顺序与编写的代码顺序可能会不一致。而为了能够让多线程并发时的顺序符合我们的预期，则需要多线程能够有序执行，需要满足happen-before原则。

happen-before原则（不指令重排）

  • 程序顺序原则：一个线程内保证语义的串行性。
  • volatile规则：volatile的写优先于读，这保证了volatile变量在多线程中的可见性。
  • 锁规则：解锁必然发生在随后的加锁前（不会将加锁治疗重排到解锁之前）。
  • 传递性：A先于B，B先于C，则A先于C。
  • 线程的start()优先于它的每一个动作。
  • 线程的所有操作优先于线程的终结。
  • 线程的终中断先于被中断线程的代码（现有线程中断，才有线程被中断）。
  • 对象的构造优先于finalize。

CAS策略

发表于 2018-09-06 | 分类于 高并发 |

通常为了保护临界区资源的安全访问，都会通过synchronized或者ReetranLock来对资源先进行加锁。这种悲观锁会导致没有获得锁的线程阻塞。将并行改为串行，大大降低了系统的吞吐量。
CAS(Compare and Swap)即比较交换。是无锁安全访问临界资源的一种策略。其假设访问临界资源不会有冲突，多线程对临界资源的访问无需加锁也无需等待阻塞。每个线程都可以访问临界区资源，也都可以对临界区资源进行修改，但修改不能提交。需要通过CAS算法来甄别是否有冲突，如果没有，则提交修改，如果有则重复当前操作直到没有冲突为止。

CAS算法

CAS参数：

  • V要更新的变量值
  • E该变量的期待值
  • N该变量的新值
    只有当 V = E 的时候，才会将V更新到N，如果V != E，则表明该变量已被修改。本次修改无法提交。

atomic包

jdk中有一个atomic包来支持CAS处理并发操作。

Unsafe

  • Unsafe提供了一系列原子性的方法来支撑CAS策略，很多数据结构例如AtomicInteger，ConcurrentLinkedQueue等，都是通过Unsafe的原子性操作来支持并发操作。
  • Unsafe底层是使用C++来实现的，并且JDK也不建议开人人员直接使用unsafe来操作。

当调用Unsafe.getUnsafe()方法来获取Unsafe的时候会对调用方的类加载器进行检查。一般应用的类加载器都是由AppLoader来加载，而系统代码的加载都是由Bootstrap来加载，通过Bootstrap来加载的时候getClassLoader会返回空，而通过AppLoader加载的非空，这样就拒绝了开发人员在代码中自行获取Unsfae。

AtomicInteger

如图是AtomicInteger的源码，可以看到实际存储数据的value和CAS策略实际实现Unsafe。

  • value字段是被volatile关键词修饰的，这样可以保证在多线程并发操作时数据的可见性。
  • valueOffset value数据位于AtomicInteger对象中的偏移量。实际对数据的获取以及修改都是基于偏移量来获取的，直接通过偏移量来处理个人理解应该是比较高效。

如图，是通过AtomicInteger来处理1000个并发线程处理将一个初始值0自增的过程。每个线程调用方法getAndIncrement来实现对临界资源进行自增。输出结果为1000显示整个并发操作的线程安全。

set(int newValue) 设置当前值

AtomicStampedReference（解决AAB问题）

CAS策略在控制并发的时候很重要的一点是将期望值与当前值进行比对。如果一致则认为没有其他线程修改，当前修改可以提交，但如果数据从A修改为B又被修改为A，实际是发生了修改但数据一致，如果业务需求需要并发控制的时候不仅仅是通过数值的一致来判断，那么就需要使用到AtomicStampedReference，即每次修改的时候不仅仅是修改value数据区域，同时还有时间戳，每次进行比对的时候也不仅仅是比对value，还要比对时间戳。只有当时间戳和value都符合期望的时候，修改可以提交。

线程安全的HashMap

发表于 2018-09-05 | 分类于 集合 |

HashTable

HashTable是线程安全的map，其对所有的get/put方法都通过synchronized关键词来保证线程安全。由于关键词synchronized修饰的是方法，所以线程在执行get/put操作时都会需要获取到这个map对象锁。效率低

Collections.synchronizedMap

如图所示，通过Collections.synchronizedMap可以将hashmap包装成一个线程安全的hashmap。

对于synchronziedMap其实也是基于synchronized来保证的线程安全性，在所有的方法操作之前都需要获取mutex对象锁。

ConcurrentHashMap

1.7版

如图所示，concurrenthashmap主要由segemtn,entry和链表构成。concurrenthashmap本质是一个segment数组，每一个segment对应着一个Entry数组，Entry数组中的每一个Entry节点都是一个单向链表的头节点。

Segment

Segment继承自ReentranLock。故Segment其实就是一个锁，Segment是其对应的Entry数组(table)的线程安全的保证。即每当要对Segment对应的table进行操作首先要对Segment加锁。

count

每一个Segment都有一个count属性，被volatile修饰的。这样每当有线程造成对table的结构性修改都会立即可见。其记录着Segment所包含的HashEntry个数，

table

每一个Segment对应着一个table，被volatile修饰。这样每当有线程造成对table的结构性修改都会立即可见。

HashEntry

HashEntry是Segment中的所有元素的抽象。其value和next都是由volatile修饰的，由于value和next都被volatile修饰，故每次线程都能读取到最新的数据，所以concurrenthashmap的读操作并没有加锁。
由于next被final修饰，故当有冲突时，节点都从链表的头插入，

concurrencyLevel

并发度就是concurrenthashmap理想可以支持并发线程数，即Segment的个数。concurrencyLevel默认是16，concurrentHashMap理想支持16个线程并发操作。

计算key所在的Segment位置

方法segmentFor就是根据key的hash值来计算key所处的Segment位置，其中SegmentShift=log(size)以2为底。segmentMask=concurrencyLevel-1。

1.8

如图所示。在JDK1.8版本中，对ConcurrentHashMap做了很大的调整。去除了Segment分段锁的设计。采用Node结构，结合单向链表和红黑树来解决冲突（当冲突链表中数据个数大于8个自动将链表转化成红黑树，查询效率有O(n)降低到O(logn)，当红黑树的节点个数小于6再转化成链表）。

put方法

通过put方法的源码可以发现：

  • 通过synchronized来对链表的头节点Node进行加锁。而没有使用1.7版本中的ReentranLock。
  • 整个concurrentHashMap的初始化是发生在第一次put的时候。
  • 指定位置节点的获取，更新以及增加的操作都是通过Unsafe利用CAS策略来实现的，这种无锁的处理方式减少了因为锁导致的线程阻塞，增大了系统的吞吐量。

get方法

通过get方法的源码，可以看出,get方法并没有加锁。

CAS操作

如图中所示，在1.8版的ConcurrentHashMap中还是使用到了CAS策略，通过Unsafe来完成获取指定位置的Node，通过比较交换来设置指定位置的Node以及增加对应位置的Node。

区别

  • 1.8版本的锁颗粒度更小，只是对Node数组中的Node节点通过synchronized来加锁。而1.7版本中是对整个Segment来加锁。
  • 1.8中除了通过链表来解决冲突之外，当链表中节点的个数大于8个之后会转换成红黑树，这样使得当有冲突时查找数据的效率由O(n)降低到了O(logn)。
