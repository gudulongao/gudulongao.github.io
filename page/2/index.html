SLF4J

SLF4J(Simple Logging Fcade Java)简单的日志门面。
SLF4J是提供了一系列的日志接口，但不提供实现，不同的日志框架来提供具体的实现，这样允许用户在最终部署系统时可以随意切换自己的日志框架。
实际SLF4J提供核心的日志API以及LoggerFactory。在使用日志的时候，不需要代码或者配置文件显示的指定日志框架，之需要通过SLF4J的api来进行日志记录，最终日志的格式，级别，输出方式等都是由具体的日志框架的配置来决定。因此可以灵活的切换日志框架而代码无感。

Logback

一个可靠、通用、快速而灵活的日志系统。性能优于log4j（大约快10倍，并更少的内存消耗）
由logback-core、logback-classic:log4j、和logback-access三个模块组成。

  • logback-core:核心代码模块
  • logback-classis:log4j:log4j改良版，同时实现了self4j的接口。支持通过sefl4j的api来使用日志系统。
  • logback-access:访问模块并与Servlet容器集成提供通过http来访问日志的功能。

配置文件的加载顺序

  1. java启动参数logback.configurationFile 指定的配置文件(如java -Dlogback.configurationFile=/path/to/mylogback.xml Test)
  2. 在classpath下查找logback.groovy文件
  3. 在calsspath下查找logback-test.xml文件
  4. 在classpath下查找logback.xml文件
  5. 如果是JDK6+，会调用ServiceLoader查找com.qos.logback.classic.spi.Configurator接口的第一个实现类
  6. 默认使用ch.qos.logback.classic.BasicConfigurator在控制台输出日志

logback会按照上面的顺序依次查找配置文件，找到了就不会再往下走（ContextInitializer#autoConfig()）。

appender

定义日志输出组件。常用的属性是name和class

属性 默认值 说明
name 无 日志输出组件的名称
class 无 日志输出组件的具体实现类，常用的有ConsoleAppender、FileAppender、RollingFileAppender
  • ConsoleAppender:控制台输出组件实现类
  • FileAppender:文件输出组件实现类
  • RollingFileAppender:结合滚动策略滚动生成日志文件的实现类

appender可以配置filter，样例配置中配置了一个ch.qos.logback.classic.filter.ThresholdFilter来过滤日志。
RollingFileAppender可以配置滚动策略（rollingPolicy），

root

定义默认的日志记录器，即针对LoggerFactory.getLogger(XXX)获取的所有Logger如果没有特殊的logger匹配，则默认就是root定义的logger。常用的属性有level

  • level 定义默认的日志记录器的日志级别

root下面可以通过appender-ref指定默认日志记录器对应的日志输出组件。样例中默认日志记录器采用控制台和滚动文件两种日志输出组件。

logger

定义日志记录器,常用的属性有name，level和additivity。

  • name 日志记录器的名称，即LoggerFactory.getLogger(name)来获取logger。通过name来匹配。同时LoggerFactory.getLogger(XXX.class)会自动转换成字符串packagepath+’.’+className+’.’+class。当然这里的name可以配置部分包路径，这样包下的所有都会匹配该logger。
  • level 日志记录器的日志级别
  • additivity 是否采用root下使用的日志输出组件(appender-ref)，日志最大容量(totalSizeCap)和日志文件最大含量(maxFileSize)，maxFileSize和totalSizeCap属性在1.1.6版本后才支持。

优势

  • 性能较log4j 快10倍，并且更少的内存消耗。
  • logback实现了self4j接口，可以为以后日志系统的切换提供便捷
  • logback除了支持xml配置还支持groovy格式的配置（更加直观简洁）
  • logback能够检测到配置文件的修改并实时加载更新配置（热部署）
  • logback能自动从I/O异常中恢复，无需重启服务
  • logback可以配置上限值，自动删除旧的日志
  • logback能够自动压缩日志
  • logback配置中支持if，then,else的条件语句，这样在dev，product等不同的环境下避免日志配置的重复
  • logback的stack trace包含更多的信息
  • logback提供更多的filter
  • logback能够结合Servlet容器提供http-access日志

Log4j2

  • 插件式结构，支持自行拓展自己的appender,logger,filter。
  • 支持json配置文件
  • 配置文件中可以直接引用属性，还可以直接替代或传值到组件

Spring Aware

Spring的依赖注入最大特点就是低侵入。对于Bean而言Spring容器是无感的。但实际应用中我们需要使用到Spring容器本身的功能，我们需要使用Spring Aware（其本身是Spring设计用来框架内部使用的），这样我们的Bean将会和Spring的代码耦合。

Aware类型 说明
BeanNameAware 获取容器中Bean的名称
BeanFactoryAware 获取容器中Bean的工厂
ApplicationContextAware 获取当前的applicationcontext
MessageSourceAware 获取message source
ApplicationEventPublisherAware 事件发生器，可以发布事件
ResourceLoaderAware 获取资源加载器，可以加载外部资源

实现相应的Aware接口就能够使用对应的Spring容器功能。接口ApplicationContext集成了ApplicationContextAware、ApplicationEventPubliserAware和ResourceLoaderAware。只要Bean实现ApplicationContext就可以获取到Spring容器的所有功能。

异步任务的执行方式

从注解@Scheduled的源码中可以看到，其支持的异步执行方式有cron，zone，fixedDelay，fixedDelayString,fixedRate,fixedRateString，initDelay和initalDelayString这几种方式。
下面简单介绍一下这几种方式。

cron表达式

cron的方式是采用Unix系统的定时任务，其表达式是由空格分隔的6（或7）个数字 秒 分钟 小时 日期 月份 星期 年（可选）组成。其具体如下表所示。

cron表达式生成网址

zone时区

cron计划任务时间所采用的时区，默认是空串，采用当前时区。样例: Asia/Shanghai

fixedDelay和fixedDelayString

上一次执行后延迟多长时间再次执行

fixedDelayString是通过字符串的方式来指定，同时支持占位符（可从配置中获取参数）

这种方式，如果定时任务执行时间长，则会等待上一个任务执行完毕之后再等待延迟时长后再次执行

fixedRate和fixedRateString

任务开始后等待多长时间后再次执行任务。
这种方式不论定时任务执行多长是否执行完毕，延时时长足够就会再次执行任务。
fixedRateString同上。

initDelay和initDelayString

第一次执行后等待时长再次执行任务。

initDelayString同上，即通过字符串的方式来注入参数，但不支持占位符（无法从配置中获取参数）

条件注解

通过@Condition条件注解，我们可以个性化的定制不同的场景来构建不同的bean。

  • 在配置类中，通过@Conditional来采用不同的条件生成不同的bean。

Spring基础

spring框架的四个特性：

  • 使用POJO进行轻量级和最小侵入式的开发。
  • 通过依赖注入和基于接口编程实现松耦合。
  • 使用AOP和默认习惯进行声明式编程。
  • 使用AOP和模版(Template)减少模式化代码。

依赖注入(DI)

所谓依赖注入是指通过容器来负责创建对象和维护对象之间的依赖关系。而不是通过对象本身负责自己的创建和解决自己的依赖。
依赖注入主要是为了解耦。体现了组合的理念。继承会使得子类与父类耦合，组合会降低耦合。

加载配置文件

注解 作用
@PropertySource 指定待加载配置文件的路径，通常用于配置类
@Value 通过注入的属性来匹配配置文件中的配置，常用于字段,即将配置文件中对应配置的值注入到当前字段

对于@Value注解，如果当前配置类中有使用注解@ProppertySource,则表明该注解匹配的配置文件是注解中指定的配置文件，否则就是加载的PeopertySourcesPlaceholderConfigurer中配置的配置文件。

正如注解的的名称，PropertySource只能加载properties配置文件，对于yml的配置，可以通过注解@ConfigurationProperties来加载。

execution表达式

切点可以通过execution表达式来声明拦截规则。
例如：execution ( com.sample.service.impl..\.*(..))

  • 第一个* 匹配返回值，表示返回类型为任意返回值。
  • com.sample.service.impl.. 匹配包路径，表明包路径com.sample.service.impl下两级目录（子孙）。
  • 第二个* 匹配类名，表示任何类名。
  • *(..) 匹配方法及方法参数，表示所有的方法参数。

args

顺序

注解 说明 入参
@before,@After 前后建言,后建言不论核心代码是否报错都会执行 JoinPoint
@Around 环绕建言。 ProceedingJionPoint（是JoinPoint的子类，其特殊拥有proceed方法）

  • 都会事先走AOP的代码。
  • 执行Around中proceed()之前的逻辑。
  • 执行Before中的逻辑。
  • 执行核心逻辑。
  • 执行Around中proceed()之后的逻辑。
  • 执行After的逻辑。
  • 如果核心逻辑成功，执行AfterReturning，否则执行AfterThrowing。

lambda表达式

发表于 2019-05-21 | 分类于 java8 |

Java8 版本的重要新特性
Lambda表达式允许将一个逻辑块作为一个方法的参数（闭包）。

语法

  • 可选类型声明：参数的类型不用声明，编译器会自动识别。
  • 可选的参数括号：一个参数可以不加圆括号，多个参数需要加括号。
  • 可选的大括号：当逻辑块只有一条语句时，无需大括号，多条语句时需要大括号
  • 可选的return关键词：当逻辑块只有一条语句时，编译器会自动根据表达式来返回返回值。多条语句需要明确返回return语句。

个人感觉这个跟js的箭头函数太像了，对于方法的参数列表，参数无需声明类型，一个参数时无需小括号，对于方法体，如果只有一句语句，无需花括号，多条执行语句才需要花括号包起来，一条语句编译器自动解析return，多条需要显示的return。好智能好方便。

样例

变量作用域

  • lambda表达式只能引用外部用final修饰的变量(即不能修改非final修饰的变量)。
  • lambda表达式中定义的变量不能与外部的变量重名。

什么是kafka

Apache kafka是消息中间件的一种。用于构建实时数据管道和流应用程序。具有横向扩展，容错，wicked fast（变态快）等优点，

名次解释

Topic

kafka将消息种子(feed)通过Topic进行分类。每一种消息的分类就是一种主题（topic）。

Producer

发布消息的一方称之为消息的生产者(producer)。

Consumer

订阅消息并处理消息的一方称之为消息的消费者（consumer）。

Broker

已发布的消息存储在kafka集群中，集群中的每一个节点都是一个kafka服务的代理，消费者从这个服务代理中拉取消息并消费。这个服务代理就是broker。

生产者消费者，生产者生产鸡蛋，消费者消费鸡蛋，生产者生产一个鸡蛋，消费者就消费一个鸡蛋，假设消费者消费鸡蛋的时候噎住了（系统宕机了），生产者还在生产鸡蛋，那新生产的鸡蛋就丢失了。再比如生产者很强劲（大交易量的情况），生产者1秒钟生产100个鸡蛋，消费者1秒钟只能吃50个鸡蛋，那要不了一会，消费者就吃不消了（消息堵塞，最终导致系统超时），消费者拒绝再吃了，”鸡蛋“又丢失了，这个时候我们放个篮子在它们中间，生产出来的鸡蛋都放到篮子里，消费者去篮子里拿鸡蛋，这样鸡蛋就不会丢失了，都在篮子里，而这个篮子就是”kafka“。
鸡蛋其实就是“数据流”，系统之间的交互都是通过“数据流”来传输的（就是tcp、http什么的），也称为报文，也叫“消息”。
消息队列满了，其实就是篮子满了，”鸡蛋“ 放不下了，那赶紧多放几个篮子，其实就是kafka的扩容。
各位现在知道kafka是干什么的了吧，它就是那个”篮子”。

特征概要

Kafka是一个分布式的流平台。一个流处理的平台所具备的3个关键能力：

1.消息的发布/订阅功能。类似消息中间件。
2.消息发生时能得到及时的处理。
3.对消息支持容错处理。

Kafka作为一个流处理平台。具备如下特征：

1.Kafka支持集群部署。
2.Kafka是以topic为类别对消息进行记录的。
3.每一条消息(record)是由key,value和ts(时间戳)组成的。

Kafka具有的4个核心API：

  • Producer API：发布一个/多个topic的消息。
  • Costumer API：订阅一个/多个topic并处理消息。
  • Sttreams API：流处理器，有效的将输入流转换成输出流。
  • Connector API：用于构建多个生产者/消费者，并将多个topic连接到现有的应用程序。

kafka的客户端和服务端的连接是基于一条简单、高性能与编程语言无关的TCP协议信道来通信

Log存储机制

kafka中的message是通过topic进行分类的。每一个topic都是相互独立的，每个topic由一个或多个partition(分区)

partition是以文件的形式存储在文件系统中的。

比如创建了一个名为page_visits的topic，其有5个partition，那么在Kafka的数据目录中(由配置文件中的log.dirs指定的)中就有这样5个目录: page_visits-0， page_visits-1，page_visits-2，page_visits-3，page_visits-4，其命名规则为[topic_name]-[partition_id]，里面存储的分别就是这5个partition的数据。

分区（Partition）

partition中存储着message，每一个message都有一个offset。是逻辑上的一个位置索引。
partition中的message由三部分组成：

部分 数据类型 说明
offset long 表示message在partition中的存储位置的偏移量，逻辑意义的位置索引
messagesize int32 表示message的大小
data kafka通讯协议规定的数据格式 消息内容

所以partition就是message按照offset顺序存储的集合。即是数据文件的集合。kafka是通过FileMessageSet来对数据文件进行操作和管理。

FileMessageSet提供了searchFile方法来从startPos位置开始查找指定的targetOffset位置的message。这个查找的逻辑是首先读取startPos位置的message，判断当前message的offset是否匹配targetOffset。如果大了，则往前读，小了再往后读。当然移动的过程中需要牵扯到messageSize。

从这里可以看到，如果数据文件很大的时候数据的查找效率很低。为了提升数据文件查找效率,kafka将数据文件做了分段和数据索引。

设计优势

采用偏移量的方式来读取消息，避免了不同的消费者对同一个分区的修改。消费者也可以更加自如的读取消息。
采用分区的方式是为了让主题可以同时处理更多的数据，也更方便支持后续的并发处理。

分段(logsegment)

kafka为了提神数据文件的查找性能。将数据文件进行了分段。

比如有100条Message，它们的offset是从0到99。假设将数据文件分成5段，第一段为0-19，第二段为20-39，以此类推，每段放在一个单独的数据文件里面，数据文件以该段中最小的offset命名。这样在查找指定offset的Message的时候，用二分查找就可以定位到该Message在哪个段中。

数据索引

对数据文件的分段，使得最终只需要在很小的数据文件中遍历查找offset对应的message了，但为了更进一步的提升查找的效率，kafka为分段后的数据文件创建了数据索引。索引与数据文件名一致，但是文件类型是index。
数据索引的每一条明细中包含两个部分：

  • 相对offset：数据分段之后，相对于当前段的offset。即message相对于分段后该段中最小的offset的相对值。比方说分段后一个段最小的offset=20,而message的offset=23，则相对offset=3。减少offset所占用的存储空间。
  • position：message在数据文件中的绝对位置。

**kafka并没有为每一条message都创建数据索引，间隔一定字节的message创建索引，这样避免索引文件占用的空间过多，方便索引可以直接加载到内存中。但这也导致没有添加索引的message无法一次定位到。还是需要基于索引的前后范围区间内进行小范围的遍历。

总结

主题page_visits有5个partition。

partition是分段的，其中某个partition中的logsegment和index如下。

我们查找offset=6的message过程如下：

  • 首先我们要找到offset=6的message位于哪个logsegment中，我们通过二分查找可以知道位于000000000000000.log中。
  • 我们000000000000000.index索引中偏移量（最小offset+相对offset）最接近目标offset的明细，我们找到了6，9807。这样我们就获取到了offset=6的message绝对位置=9807。
  • 我们直接从000000000000000.log中从9807开始读就能读取到想要的message。

分布式

上面提到的kafka中topic的partition，是可以分布式的存储在多个kafka节点中的。这样每个节点只需要处理其分到的partition。当然还可以通过配置将partition的数据复制到别的节点用于容错。当一个partition有多个节点负责存储处理的时候，这些个节点之间会有leader和follower两个角色。leader负责处理这个分区所有的读写操作请求。而follower负责同步leader节点的数据。如果leader宕机，其余的follower节点可以立即重新推举新的leader节点。同时kafka中的一个节点可以是一个partition的leader，也可以是其他partition的follower。这样做避免所有的请求都只让一个节点处理，达到负载均衡的效果。

生产者(Producer)

生产者负责将message发布到topic的partition上。最简单的是从partition列表中依次轮流的发布。也可以根据某种权重算法来选择。

消费者(Consumer)

消费模型有队列和订阅两种模型。

类型 说明 优点 缺点
队列 多个消费者共享一个topic，但是一条message只能被一个消费者消费 可以通过增加消费者来提升消息处理的能力，提升吞吐量 一旦消费了消息的消费者进程宕机，消息就彻底丢失
发布/订阅 多个消费者共享一个topic，一条消息会广播给多个消费者，收到消息的消费者都可以处理消息 消息可以同时被多个消费者处理 由于会广播消息，所以无法收缩消息的处理

消费组

但是在kafka中，专门抽象了一个消费组的模型。当多个消费者在同一个消费组内时，发布在topic的消息只会发给消费组中一个消费者，即队列模式。当多个消费者都位于不同的消费组时，消息会广播到不同的消费组，即发布/订阅模型。

生产者发布message时会指定topic中的partition。kafka能够保证生产者发布的消息能够按照发布的顺序顺序加入到topic的partition中。并且这个消息只会被消费组中的一个消费者消费。即保证了partition层面的消息能够顺序的发布，同一个topic的partition中的消息能够顺序处理。
一个消费组的消费者数量不要多过partition的数量，避免有空闲的消费者

Kafka的保证

  • 生产者将message发布到topic的partition中，kafka能够保证message会按照生产者发布的顺序加入到partition中。同时，消费者也会按照顺序收到message。即producer顺序发布的消息m1和m2。则m1的offset会比m2小。
  • 如果topic配置了复制因子N，则N-1个节点宕机也不会丢失已提交的message。上面提到的消费组，kafka能够保证同一个partition中的消息被一个消费组中的一个消费者消费。故能够保证partition内的消息能够被顺序消费。

存储系统

kafka本身也是一个高性能，低延迟的分布式文件系统。
写入到kafka的数据将写入到磁盘并复制到集群中保证数据的容错性。

流处理

kafka能够支持实时的流处理。可以持续的获取输入topic的数据，加工处理后写入输出topic。通过stream API可以构建聚合计算的复杂应用程序。

适用场景

消息中间件

kafka作为消息中间件可以接耦数据的生产和消费场景，与传统的消息中间件拥有更高的吞吐量，分区，副本和故障转移更利于大数据量的消息处理。

网站活动追踪

kafka原本的使用场景：用户的活动追踪，网站的活动（网页游览，搜索或其他用户的操作信息）发布到不同的话题中心，这些消息可实时处理，实时监测，也可加载到Hadoop或离线处理数据仓库。
每个用户页面视图都会产生非常高的量。

监控

可以将分布式应用的数据进行统计汇总聚合。

日志聚合

许多人使用Kafka作为日志聚合解决方案的替代品。日志聚合通常从服务器中收集物理日志文件，并将它们放在中央位置（可能是文件服务器或HDFS）进行处理。Kafka抽象出文件的细节，并将日志或事件数据更清晰地抽象为消息流。这允许更低延迟的处理并更容易支持多个数据源和分布式数据消费。
