重入锁(ReentrantLock)

实现接口Lock。显示进行加锁，释放锁的操作使得并发处理时更加的灵活。

同步代码块（lock/unlock）

如图是使用重入锁来进行并发操作的样例，可以看到两个线程都共享了重入锁lock，在同步代码块的开始需要手工显示的进行加锁。在同步代码块的最后，需要手工显示的释放锁。
重入锁名称的缘由是指在同一个线程内，可以多次进行加锁，同时，最终也要对等的多次进行释放锁。

响应中断(lockwithinterruptibly)

重入锁相比synchronized，在线程处理临界资源的时候，除了获取锁继续执行和阻塞之外，使用重入锁还有个优势就是可以支持中断。即在未获取到临界资源导致线程阻塞的时候，可以通过响应中断来停止阻塞的状态。这种响应中断的方式对于解决死锁有很大的帮助。

方法名 说明
void lockInterruptibly() 支持响应中断的加锁

上面的样例就是营造一个死锁的场景，FirstThread运行后会获取readlock，成功后又想获取writelock。而AfterThread运行后会先获取writelock，成功后想获取readlock。这个时候两个线程就发生了死锁。两个线程都无法完成自己的工作。这个时候对FirstThread进行了中断，在FirstThread处理中断异常的时候，释放了所拥有的所有资源，而AfterThread便成功获取了readlock而完成工作。
重入锁的lockWithInterruptibly方法会有可能抛出InterruptedException异常，所以可以在catch里能够专门处理中断时的场景。

限时等待(trylock，解决死锁)

为了避免死锁，除了支持中断的方式之外，重入锁还提供了这种限时等待的方式来避免发生死锁。

方法名 说明
boolean tryLock() 尝试获取锁，如果能够获取，立即返回true，否则返回false
boolean tryLock(long timeout,TimeUnit unit) 尝试获取锁，如果不能立即获取，会等待指定的时长，最后再返回获取的结果

根之前的样例类似，每个线程都是在不断的循环直到完成工作，FirstThread会先获取lock1的锁，再获取lock2的锁。如果没有能成功，则在finally里释放锁。AfterThread的内容一致只是获取锁的顺序不一样，先获取lock2的锁再获取lock1的锁。如果都不成功则释放所有的锁。因为两个线程获取到锁以后的等待时长是不一致的，所以总会有一个线程释放所有资源之后另个线程能够拿到所有所需的资源继续执行。这样就不会导致死锁发生。
重入锁提供trylock方法来解决死锁问题，无参的trylock表示立即获取锁，如果成功则返回true，否则返回false，不会阻塞等待，有参的trylock有有两个参数，第一个是时长，第二个是时长的单位，指如果没有获取到锁会阻塞等待的时长，不会像synchronized那样一直阻塞到获取到锁。

公平锁（解决饥饿）

线程运行的时候因为线程的优先级会导致优先级高的线程总会有更高的概率获取临界资源。就会导致优先级的线
程总示无法获取临界区资源从而产生饥饿现象。

方法名 说明
void ReentrantLock(boolean fair) 构建一个采用公平锁策略的重入锁

重入锁的构造可以注入是否采用公平锁，如果true则表明所有的线程在获取锁的时候按照申请的时间来排序，即先申请获取锁的线程先获取。

缺陷

公平锁需要额外的维护一个有序队列来保证线程申请锁的顺序。实现成本高，性能不如默认的非公平锁。

等待和继续（Condition）

Object对象的实例方法wait/notify与synchronized结合，可以控制线程等待和继续执行。重入锁可以结合Condition的await/signal达到一样的效果。

  • Condition是从重入锁而来。
  • 与Object的wait和notify一样，使用condition的await/signal的时候，也必须先获取重入锁。

    方法列表

方法名 说明
await() 等待，将当前线程加入到condition的等待线程队列
awaitUninterruptibly() 等待，但无法响应中断
boolean await(long timeout,TimeUnit unit) 有限时长的等待
signal() 从condition的等待线程队列中随机去除一个线程
singalAll() 清空condition中等待线程队列的所有线程

与synchronized的区别

  • synchronized不支持中断，即使用synchronzied处理并发的时候，线程只有两种状态：要么拿到锁成功执行，要么等待阻塞直到获取到锁。而对于重入锁，可以通过lockwithinterruptibly来支持中断响应，也可以通过trylock来限时等待锁。
  • synchronized容易产生死锁，即获取不到锁线程会一直阻塞也不释放资源。而重入锁在lockwithinterruptibly方法支持中断响应，即在获取不到锁阻塞的时间，会抛出InterruptException异常，我们可以在异常中释放资源，这样避免死锁的发生，甚至可以通过trylock的方式来获取锁，限时等待锁的时长甚至根本不等待。
  • synchronized结合Object对象的实例方法wait/notify来让线程等待/恢复执行。重入锁结合Condition的await/signal来让线程等待/恢复执行。
  • synchronzied是非公平锁，重入锁默认是非公平锁，支持公平锁。
  • 在JDK1.8版本之前，synchronized在高并发场景下的效率是不如ReentrantLock的。在1.8版本后JDK对synchronized做了优化，目前效率相差不大。

信号量（Semaphore）

synchronzied和ReentrantLock都限制在同步代码块中只能有一个线程。而信号量的作用是允许同步代码块中的线程可以有多个。

运行结果如图：

线程的运行逻辑中运用到了信号量，信号量的令牌总共有3个，而实际线程池中的线程有20个，但是提交任务提交了12个，即其实整个线程池是可以满足12个任务由12个线程一次并发执行的，但是由于我们的信号量机制，总共只有三个令牌，所以通过输出的结果可以看出，同一时间，只有三个线程在完成工作，所以信号量保证了同步代码块中的线程数量可以多于一个线程。

方法列表

方法名 说明
Semaphore(int permits) 构造，构造一个拥有permits个令牌的信号量
Semphore(int permits,boolean fair) 同上，支持公平锁
acquire() 申请令牌，如果有空余令牌，则继续执行，否则会阻塞等待
acquireUninterruptibly() 同上，不处理中断响应
boolean tryAcquire() 尝试获取令牌，没有空余令牌返回false，获取成功返回true
boolean tryAcquire(long timeout,TmitUnit unit) 尝试获取令牌，没有空余则等待timout时长，还没有则返回false，否则返回true
release() 返回令牌

对于信号量来处理并发的时候，在获取令牌之后一定要记得relase，否则令牌会越来越少。

读写锁(ReadWriteLock)

基于并发级别中的RCU(Read Copy Update)可以看到如果可以将读写分离，对于系统的吞吐量会有很大的提升。
concurrent包中提供ReentrantReadWriteLock（实现接口ReadWriteLock）来将读写分离，通过ReentrantReadWriteLock可以分别获取读锁和写锁，这样将读写分离。

样例中，各有18个读线程和写线程，写线程共享writelock，读线程共享readlock。每个线程运行时都会输出系统时间。我们可以看下输出。

  • 读线程来自于ReentrantReadWriteLock的实例方法getReadLock，写锁来自于实例方法getWriteLock。
  • 18个写线程每个线程执行的时候时间都是不一样的，即每个写操作共享写锁的时候，都是互斥的，会阻塞。
  • 18个读线程每个线程的时间都一样，即每个读线程共享读锁的时候，并不互斥。

计数器（CountDownLatch）

对于多线程并发执行任务时，需要当所有任务都执行完毕后再继续执行工作，这种时候就需要使用到CountDownLatch。

  • 样例中有10个并发执行的多线程任务。CountDownLatch构造的时候初始化值也是10
  • 每个多线程任务都共享CountDownLatch对象。并且每个线程完成任务之后调用方法countDown，来告知计数器已有一个线程完成任务。计数器会减一。
  • 主线程会通过CountDownLatch的awati方法阻塞，直到计数器倒数到0即所有多线程任务都完成的时候，主线程才会继续执行。

循环栅栏（CyclicBarrier）

方法名 说明
CyclicBarrier(int parties, Runnable barrierAction) 构造函数，指定需要等待的线程数，指定栅栏动作
int getParties() 获取需要等待的线程数,即构造函数指定的parties
int getNumberWaiting() 获取还在等待的线程数
int await() 阻塞当前线程,返回值为在栅栏的等待线程队列中的位置索引
int await(long timeout, TimeUnit unit) 有限时长的阻塞线程，指定时长，指定时长单位

先上样例代码

  • 循环栅栏的构造函数有两个参数，一个是指定需要栅栏拦截停止的线程数，另一个是当拦截停止的线程数足够的时候，执行的栅栏任务。
  • 多线程任务需要共享同一个栅栏对象，每当多线程执行到栅栏的await方法时，就会进入栅栏的等待线程队列中，同时栅栏的计数器会减1。这样多线程任务会暂停，直到暂停的多线程数量符合预期之后，栅栏会执行栅栏任务。
  • 当栅栏任务执行完毕之后，栅栏会将其等待线程队列中的所有线程都恢复运行，同时计数器的计数也会重置到初始值，方便下次多线程任务进行await的时候时候。

书中的例子很生动，多线程任务就等同于点兵，栅栏任务即点兵符合点兵的数目之后执行的任务。比方说某项任务，需要集齐10个士兵上车，人员整齐之后开往目的地，10个士兵再下车，都下车之后，布置任务。10个士兵就是多线程任务，10个士兵都并发的上车，上车之后执行栅栏的await方法，暂停休息等待10个士兵都上车，即线程都await之后，人齐了，等待栅栏任务即开往目的地，等到达目的地之后10个任务重新被激活，开始并发的下车，下车之后再次调用栅栏的await方法，暂停休息等待所有士兵都下车即10个线程都await之后，等待栅栏任务即分配任务。

注意

栅栏的计数器会导致只有当拦截的线程数达到预定的数目之后才会执行栅栏任务，也只有执行栅栏任务之后暂停的多线程任务才会被重新激活继续运行，如果多线程任务数量小于栅栏预定的拦截线程数目，那么整个流程都会卡在这里。如果多线程任务的数量大于栅栏预定拦截的线程数，则栅栏计数器的功能就整个失效了，整个流程拦截暂停多线程任务的效果就呈现一种混乱的状态。栅栏预定拦截的多线程任务数量一定要与多线程任务数量一致。

阻塞工具(LockSupport)

线程阻塞的工具类，其提供静态的方法park来阻塞线程，提供静态的方法unpark来唤醒线程。

LockSupport有以下几个优点：

  • LockSupport基于信号量为1的机制，其为每一个线程都准备了一个令牌。执行park方法的时候，如果有令牌可用，则不会阻塞，如果令牌不可用才会阻塞，只有unpark方法会使得令牌可用。即如果先执行了unpark方法，也不会使得线程永久的挂起，因为只要执行过unpark方法，就会有令牌可用，这个时候再执行park方法并不会阻塞。
  • 不会抛出InterrupteException异常，但是LocSupport的park方法是支持中断响应的，即可以在调用park方法后，通过线程的中断标志来判断是否有中断，如果有则处理中断场景。

与Suspend/resume的区别

  • suspend/resume方法是Thread的实例方法，现已被标示为废弃方法。park和unpark方法为LockSupport的静态方法。
  • suspend无法处理中断。而park方法可以通过线程的中断标志来处理中断。
  • 一旦resume执行在suspend之前，suspend后面没有resume的话，线程会被永久挂起。而LockSupport则不会，基于信号量机制，哪怕unpark优先于park执行，park遍不会阻塞挂起，会立即返回。
