重入锁

重入锁(ReentrantLock)是完全可以替代synchronized关键词的。在JDK1.5早期的版本之前，重入锁的性能远远高于synchronized。JDK1.6版本之后对synchronized做了大量的优化，现在两个锁的性能相差不大。

重入锁的特点

手工加锁，手工释放

重入锁之所以叫重入，是因为同一个线程内，可以对同一个对象锁多次加锁。但是相应的也要多次释放锁。

可重入

ReentrantLock是支持可重入的，即同一个线程对同一个对象可以重复加锁。同样的，释放锁时也需要释放对应的次数。

释放的次数(releaseNum)与加锁(lockNum)的次数不一致的问题：

  • releaseNum<lockNum，释放的次数小于加锁的次数，则对象锁资源还是没有得到释放，其他线程也就永远无法获取到锁资源而会造成其他鲜橙一直阻塞。
  • releaseNum>lockNum，释放的次数大于加锁的次数，则会抛错IllegalMonitorStateException。

支持公平锁

多线程高并发的场景下，线程间的执行顺序是不确定的，优先级的不同还会造成线程的饥饿，即有些线程总是阻塞无法进入临界区。
公平锁的意义在于解决饥饿，让所有的线程都排队执行，先来的线程先执行。让所有的线程都能有机会进入临界区运行。

公平锁的实现是内部有一个队列来维护线程的先进先出的规则，所以效率较低，在没有特别需要解决饥饿的场景下不建议使用！

API

方法名 说明
void await() 当前线程进入condition的阻塞队列，直到别的线程通过condition调用notify/notifyAll有可能重新获取到锁后继续执行，支持中断响应

void signalAll() 唤醒condition等待队列中的所有线程，待当前线程释放锁资源之后，只有成功竞争到锁的线程才能够继续执行

线程池

线程池的背景

  • 线程的创建和销毁需要花费时间，如果每一个小任务都需要创建线程，很有可能创建和销毁线程锁占用的时间会比实际任务执行的时长更长。得不偿失。
  • 线程的使用必须要控制在一个范围，线程本身也要瓜分内存空间，大量的线程会抢占宝贵的内存资源。过多会导致OutOfMemory异常。同时大量的线程也会增加GC的停顿时间。
  • 有了线程池，就避免了系统频繁的创建和销毁线程，可以让创建的线程进行复用，通过线程池保持几个活跃的线程。这样创建线程的过程变成从线程池获得空闲线程，关闭线程变成将线程归还给线程池。

自带线程池

JDK的concurrent包Executors框架提供了以下几种线程池

FixedThreadPool

Execotors.newFixedThreadPool():线程池中的线程数是固定的。当有一个新的任务提交时，如果有空闲线程，则立即执行，如果没有空闲线程，任务存放到任务队列中等待有空闲线程后再执行。

可以看出，创建FixedThreadPool时，核心线程数和最大线程数是一样的。并且任务队列采用了LinkedBlockingQueue。
